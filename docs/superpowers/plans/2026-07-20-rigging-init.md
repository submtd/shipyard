# rigging increment 1 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build rigging v0.1.0 — a stdlib-only pure engine that authors one injection-safe GitHub Actions test workflow per repo (Python + Node), plus a `rigging:init` skill that detects the stack, proposes `.rigging.json`, and scaffolds the workflow with absolute no-clobber.

**Architecture:** Mirror keel: a pure engine (`config → detect → plan → render`, no subprocess/IO, enforced by an AST purity test) driven by a pure-data stack registry; the `rigging:init` skill gathers signals via bash and calls the pure functions. No `templates/` dir — `render()` over the registry is the single source of truth; the drift guard is a dogfood test (`render(shipyard config) == committed ci.yml`).

**Tech Stack:** Python 3.9+, stdlib only. pytest.

## Global Constraints

- **Runtime dependencies: stdlib only.** Tests may use pytest.
- **Python 3.9+.** Any module using `X | None` annotation syntax MUST begin with `from __future__ import annotations`. No `match`; no `X | Y` outside an annotation context.
- **The engine is pure.** `config.py`, `stacks.py`, `detect.py`, `plan.py`, `render.py`, `scaffold.py` MUST NOT import `subprocess`, `os`, or networking modules. Filesystem reads use `pathlib` only. IO and interaction live in the skill. A `test_purity.py` (copied from keel) enforces this via AST.
- **Injection-safe by construction.** `render()` has no code path that interpolates a `${{ ... }}` expression into a `run:` block. The only expression emitted anywhere is `${{ matrix.<var> }}` inside a `with:` input. No `github.*` context anywhere in increment 1.
- **`.rigging.json` keys are camelCase.** `config.load_config(root) -> Optional[Config]` returns `None` when the file is absent and raises `ConfigError` on any invalid-but-present file (mirrors `keel.config.load_config`).
- Skill frontmatter uses **only** `name` and `description`; `name` MUST equal the directory name (`init`).
- Work on branch `feature/rigging` (already created off `main`). Do NOT switch branches. Commit with `git -c user.name="Steve Harmeyer" -c user.email="harmeyersteve@gmail.com"`.
- This is a new plugin → it ships as version **0.1.0**.

### Task 2: `config.py` — `.rigging.json` loader/validator

**Files:** Create `plugins/rigging/rigging/config.py`; Test `plugins/rigging/tests/test_config.py`.

**Interfaces:**
- Consumes: `stacks.STACK_IDS`.
- Produces:
  - `config.ConfigError(Exception)`.
  - `config.Config` — frozen dataclass: `name: str`, `stacks: dict[str, tuple[str, ...]]` (ordered: stack id → resolved version tuple).
  - `config.load_config(root) -> Optional[Config]` — returns `None` if `.rigging.json` absent; raises `ConfigError` on any invalid value; else a `Config`.

**Validation rules (exact):**
- Read `<root>/.rigging.json`; unreadable/invalid JSON or non-object → `ConfigError`.
- `name`: optional, default `"ci"`; must be a str matching `^[A-Za-z0-9_-]+$` else `ConfigError` (this is the output-filename stem — it must not contain `/`, `.`, or `..`).
- `stacks`: required; must be a non-empty object else `ConfigError`. Iterate in JSON key order (Python 3.7+ dict preserves it). Each key must be in `stacks.STACK_IDS` else `ConfigError` naming the bad id and listing allowed ids. Each value must be `null` or an object; if an object it may have `versions`: a non-empty list of strings, each matching `^[A-Za-z0-9][A-Za-z0-9._+-]*$` (rejects whitespace/newline/`}}` → closes YAML-structure injection) else `ConfigError`. Omitted/`null` versions → the stack's `stacks.REGISTRY[id].default_versions`.
- `Config.stacks` preserves key order and holds resolved version tuples.

### Task 5: `render.py` + golden fixtures

**Files:** Create `plugins/rigging/rigging/render.py`, `plugins/rigging/tests/golden/{python,node,polyglot}.yml`; Test `plugins/rigging/tests/test_render.py`.

**Interfaces:**
- Consumes: `plan.CiPlan`, `plan.Job`, `stacks.Step`.
- Produces:
  - `render.render(plan: plan.CiPlan) -> str` — deterministic GitHub Actions YAML.
  - `render.iter_run_blocks(yaml_text: str) -> list[str]` — returns every `run:` step's body text (used by the security test and future hook).

**Emitter rules (exact):**
- Top of file:
  ```yaml
  name: <plan.name>
  on: [push, pull_request]
  permissions:
    contents: read
  jobs:
  ```
- One job per `Job`, keyed by `job.id`:
  ```yaml
    <id>:
      runs-on: "ubuntu-latest"
      strategy:
        matrix:
          <matrix_var>: ["3.9", "3.12"]
      steps:
        - uses: "actions/checkout@v4"
        - uses: "actions/setup-python@v5"
          with:
            python-version: "${{ matrix.python }}"
        - run: "pip install pytest"
        - run: "python -m pytest"
  ```
- **Double-quote every scalar VALUE** (runner, action refs, version list items, `with:` values, `run:` bodies) so `"3.10"` never coerces to the float `3.1`. Structural keys are unquoted. The matrix version list renders as a flow sequence of quoted strings: `["3.9", "3.12"]`.
- The ONLY `${{ ... }}` emitted is the `with:` version value `"${{ matrix.<var> }}"`. `run:` bodies are `Step.run` verbatim (already literal, no expressions).
- Deterministic: same plan → byte-identical output every call.
- `iter_run_blocks` parses the emitted text for `- run: "<body>"` lines (single-line runs only in increment 1) and returns the unquoted bodies.
