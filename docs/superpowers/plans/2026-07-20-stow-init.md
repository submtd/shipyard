# stow increment 1 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build stow v0.1.0 — a stdlib-only pure engine with a live managed-block splice that idempotently writes AND updates a repo's `.gitignore` from a data registry (`base`/`python`/`node`), plus a `stow:init` skill, never destroying user-custom lines.

**Architecture:** Mirror rigging: pure engine (`config → detect → desired_sections → blocks`), AST-purity-guarded, driven by a pure-data section registry. The heart is `blocks.py`, a pure string→string managed-block engine where `create == apply_blocks("", desired)`.

**Tech Stack:** Python 3.9+, stdlib only. pytest.

## Global Constraints

- **Runtime dependencies: stdlib only.** Tests may use pytest.
- **Python 3.9+.** Any module using `X | None` annotation MUST begin with `from __future__ import annotations`. No `match`; no `X | Y` outside an annotation context.
- **The engine is pure.** `stacks.py`, `blocks.py`, `config.py`, `detect.py`, `scaffold.py` MUST NOT import `subprocess`, `os`, or networking modules. Filesystem reads use `pathlib` only. An AST `test_purity.py` (mirroring rigging/keel) enforces this.
- **Parser integrity.** No registry body line may contain a newline or match the stow marker regex — so stow can never emit a body that breaks its own parser. Pinned by a test.
- **Managed-block safety.** `apply_blocks` never touches a line outside a well-formed stow region; raises `StowError` on a malformed region (opener without closer, duplicate openers for one id) rather than guessing. Idempotent: `apply_blocks(apply_blocks(x)) == apply_blocks(x)`.
- **`.stow.json` keys are camelCase.** `config.load_config(root) -> Optional[Config]` returns `None` when absent, raises `ConfigError` on any invalid-but-present file.
- Skill frontmatter uses **only** `name` and `description`; `name` MUST equal the directory name (`init`).
- **No `plugins/stow/tests/__init__.py`** — a package-named tests dir shadows same-basename sibling test files under `--import-mode=importlib`. A guard test in the suite already forbids `plugins/*/tests/__init__.py`.
- Work on branch `feature/stow` (already created off `main`). Do NOT switch branches. Commit with `git -c user.name="Steve Harmeyer" -c user.email="harmeyersteve@gmail.com"`.
- This is a new plugin → version **0.1.0**.

### Task 2: `blocks.py` — the managed-block engine (the heart)

**Files:** Create `plugins/stow/stow/blocks.py`; Test `plugins/stow/tests/test_blocks.py`.

**Interfaces:**
- Consumes: `stacks.StackSpec`.
- Produces:
  - `blocks.StowError(Exception)`.
  - Marker constants + compiled regexes: opener `^# >>> stow:(?P<id>[a-z0-9-]+) >>>$`, closer `^# <<< stow:(?P=id) <<<$` (full-line, anchored). Expose the "is any stow marker line" pattern for the Task 1 parser-integrity test to reuse.
  - `blocks.render_block(spec: stacks.StackSpec) -> str` — deterministic block text: opener line, exactly one fixed advisory comment line (e.g. `# managed by stow — edits inside this block are overwritten; put custom entries outside it`), the body lines, closer line. No trailing blank line inside.
  - `blocks.find_blocks(text: str) -> tuple` — returns `(well_formed, malformed)` where `well_formed` is a list of `(id, start_line_idx, end_line_idx)` for matched opener/closer pairs and `malformed` flags unmatched/duplicate markers. Used by both `apply_blocks` and the dogfood.
  - `blocks.apply_blocks(existing_text: str, desired_sections: list) -> str` — `desired_sections` is an ordered list of `StackSpec` (base first, then registry order). Implements the splice:
    1. Parse existing into lines. Identify well-formed stow regions; on any malformed region raise `StowError` naming the line.
    2. For each desired section present as a region → replace that region (opener..closer inclusive) in place with `render_block(spec)`.
    3. For a known-registry region NOT in desired → drop it (and collapse surrounding blank lines cleanly).
    4. For an unknown-id region → leave untouched.
    5. For each desired section with no existing region → append at end in canonical order, separated by exactly one blank line.
    6. Lines outside any region are emitted verbatim in original order. Guarantee exactly one trailing newline. Normalize input line endings to `\n`.
  - **Invariant:** `apply_blocks("", desired)` produces the from-scratch file (so `create == apply_blocks("", ...)`), and `apply_blocks(apply_blocks(x, d), d) == apply_blocks(x, d)`.

- [ ] **Step 1: Write failing tests** — `test_blocks.py` (this is the largest test file; cover every rule):
  - `create == apply_blocks("", [BASE, python])` equals a known from-scratch expected string; `render_block` is deterministic and contains the advisory comment + body + markers.
  - append when a section is absent; replace in place when present (put a stale body in a block, apply, confirm it becomes the current body at the same position).
  - FREE lines preserved verbatim before/between/after blocks (e.g. a `.superpowers/` line and a comment survive untouched, in place).
  - idempotency: `apply_blocks(apply_blocks(x, d), d) == apply_blocks(x, d)`.
  - declarative removal: a `stow:node` block present but node not desired → block removed; a `stow:python` desired → kept/updated.
  - unknown-id region (`# >>> stow:rust >>>`…`# <<< stow:rust <<<`) left untouched even when not desired.
  - malformed: opener without closer → `StowError` naming the line; duplicate openers for one id → `StowError`.
  - multi-stack composition order = base then registry order; exact single-blank-line separators; single trailing newline; input with `\r\n` normalized to `\n`.
- [ ] **Step 2: Run to verify fail.**
- [ ] **Step 3: Implement** `blocks.py` (pure; `re` + string ops only).
- [ ] **Step 4: Run** test_blocks green, full suite green.
- [ ] **Step 5: Commit** `feat(stow): managed-block splice engine (apply_blocks, idempotent, user-lines-safe)`.

### Task 6: the `stow:init` skill

**Files:** Create `plugins/stow/skills/init/SKILL.md`.

Mirror `rigging:init` (read it). Frontmatter EXACTLY `name: init` + `description`. Flow:
1. Confirm repo root (`git rev-parse --show-toplevel`).
2. Check `.stow.json` up front (3-way): absent → fresh-scaffold; present-and-loads → use it as-is; present-but-`ConfigError` → stop, surface error, do not write.
3. Fresh-scaffold: `detect_stacks` (via `python3 -c` with `sys.path.insert(0, "${CLAUDE_PLUGIN_ROOT}")`); if none detected, ask which of python/node (or base-only); `propose_config`; show `.stow.json`; confirm; exclusive-create (`open(path,"x")`) `.stow.json`.
4. Apply to `.gitignore`: `apply_blocks(existing_text_or_empty, desired_sections(load_config(".")))` and write the result. Because `create == apply_blocks("", ...)`, create-if-absent and splice-if-present are the SAME call — read the file if it exists (else `""`), apply, write back. This is a managed merge, NOT no-clobber: user lines outside the markers are preserved; re-running is the update.
5. Verify idempotency: re-apply and assert the text is unchanged; assert no malformed markers. Report what changed, and note custom lines belong OUTSIDE the markers. Point at keel:init / rigging:init for the sibling layers.
